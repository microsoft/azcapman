# T-2.6 — Red-team audit of Catalogs API migration

## Goal

Adversarial review of all changes from T-2.1 through T-2.5 to verify correctness, consistency, and absence of regressions.

## Why

The migration touches initialization order, adds new functions, modifies existing functions, and introduces a new parameter across 3 scripts. A red-team audit catches inconsistencies, missed edge cases, and contract violations.

## Scope

All files modified by T-2.1 through T-2.5.

## Acceptance criteria

1. Initialization order is identical across all 3 scripts (subscriptions → locations → SKUs)
2. `Get-VMCatalogData` function signatures are consistent across all 3 scripts (allowing for az CLI vs Az PowerShell differences)
3. Pagination implementation is correct (nextLink loop terminates, doesn't infinite-loop)
4. `Get-SKUDetails` fallback behavior is identical across all 3 scripts
5. `$CatalogLocation` parameter exists with default `eastus` in all 3 scripts
6. No references to the old initialization order remain
7. No `Az.Reservations` or `az reservations` dependencies introduced
8. Return type of `Get-SKUDetails` remains `string[]` in all 3 scripts
9. All 3 scripts parse without PowerShell syntax errors
10. Zero stale references to `AutofitComboMeterData` or `NormalizedSKU` or `costmanagementconnector-data` in source files
